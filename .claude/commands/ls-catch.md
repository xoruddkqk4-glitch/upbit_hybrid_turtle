# ls-catch: 이전 작업 이어서 하기 커맨드

새 대화를 시작했을 때, 이전에 어디까지 작업했는지 한눈에 파악하고 바로 이어서 작업할 수 있도록 도와준다.
코딩을 모르는 사용자도 현재 상태를 완전히 이해할 수 있도록 쉬운 말로 설명한다.

---

## 진행 순서

### 1단계 — 작업 중인 계획서 찾기

`.claude/plans/` 폴더에서 계획서 파일을 모두 찾는다.

- 파일이 없으면:
  ```
  현재 진행 중인 계획서가 없습니다.
  새 작업을 시작하려면 /ls-plan 을 사용하세요.
  ```
  라고 안내하고 종료한다.

- 파일이 여러 개면: 가장 최근에 수정된 파일을 주 계획서로 선택한다.

계획서를 찾으면 아래 내용을 파악한다:
- 계획서 파일명과 목표(🎯 목표 섹션)
- 전체 진행 현황 표(📌 전체 진행 현황 섹션) — 어떤 단계가 ✅ 완료이고 어떤 단계가 ⬜ 남아 있는지
- 현재 단계(파일 상단 `> 현재 단계:` 항목)

---

### 2단계 — git 작업 현황 확인

아래 명령어를 실행해서 git 상태를 파악한다:

- `git log --oneline -10` — 최근 커밋 10개 (무슨 작업이 저장됐는지)
- `git status` — 현재 변경되었지만 아직 저장 안 된 파일 목록
- `git diff HEAD` — 저장 안 된 변경 내용 상세

---

### 3단계 — 현황 보고

아래 형식으로 현재 상태를 한눈에 보고한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 이전 작업 현황 보고
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📁 작업 중인 계획서
  파일: [계획서 파일명]
  목표: [계획서의 목표를 쉬운 말로 한 줄 요약]

─────────────────────────────────────
✅ 지금까지 한 작업
─────────────────────────────────────

[완료된 단계 목록 — 단계 이름과 한 줄 설명]
예:
  ✅ P1-1 자료 수집 — 라이브러리 기능, API 경로 확인
  ✅ P2-1 설계안 작성 — 8개 파일 구조 설계 완료
  ...

─────────────────────────────────────
📝 수정된 파일 (아직 저장 안 됨)
─────────────────────────────────────

[git status 결과를 쉬운 말로 설명]

변경 없으면:
  "현재 저장하지 않은 변경 내용이 없습니다."

변경 있으면:
  새로 생긴 파일:
    + 파일명 — 한 줄 설명
  수정된 파일:
    ✏️ 파일명 — 무엇이 바뀌었는지 한 줄 설명

─────────────────────────────────────
💾 최근 저장(커밋) 내역
─────────────────────────────────────

[git log 결과 — 커밋 ID와 메시지를 쉬운 말로 설명]
예:
  · eef186a — 자동매매 봇 전체 파일 최초 생성 (2026-04-12)
  · a1b2c3d — 트레일링 스탑 버그 수정 (2026-04-11)

─────────────────────────────────────
👉 다음에 할 작업
─────────────────────────────────────

[계획서에서 ⬜ (미완료) 중 첫 번째 단계를 찾아 설명]
예:
  다음 단계: P4-1 단위 테스트
  할 일: 코인 필터, 감시 리스트, 주문 함수가 각각 혼자서 잘 작동하는지 확인

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

바로 이어서 진행할까요? (네 / 아니오)
```

---

### 4단계 — 이어서 작업 시작

사용자가 **'네'** 라고 하면:
- 다음 단계를 즉시 시작한다
- ls-plan 커맨드의 진행 방식(단계별 보고·승인)을 그대로 따른다

사용자가 **'아니오'** 라고 하면:
- 다음 작업을 어떻게 진행할지 사용자가 원하는 방향을 물어본다

---

## 계획서가 여러 개일 때

`.claude/plans/` 폴더에 계획서가 2개 이상 있으면:

```
진행 중인 계획서가 여러 개 있습니다. 어떤 작업을 이어서 할까요?

1. 20260412-나스닥골든크로스자동매매.md — 현재 단계: P4-1 단위 테스트
2. 20260410-텔레그램봇만들기.md — 현재 단계: P3-2 구현
3. 20260408-백테스트시스템.md — 현재 단계: P2-3 코드 구조 작성

번호를 입력하거나, 그냥 '1번' 처럼 말씀해 주세요.
```

사용자가 선택하면 해당 계획서를 기준으로 3단계 현황 보고를 진행한다.

---

## 계획서 없이 git만 있을 때

`.claude/plans/` 폴더가 없거나 비어 있어도, git 저장소가 있으면 git 정보만으로 보고한다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 이전 작업 현황 보고
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📁 작업 중인 계획서
  계획서 파일이 없습니다.

─────────────────────────────────────
💾 최근 저장(커밋) 내역
─────────────────────────────────────

[git log 결과]

─────────────────────────────────────
📝 수정된 파일 (아직 저장 안 됨)
─────────────────────────────────────

[git status 결과]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

새 작업을 시작하려면 /ls-plan 을 사용하세요.
```

---

## 절대 규칙

1. **설명은 항상 쉬운 말로** — 기술 용어가 필요하면 바로 옆에 쉬운 설명을 덧붙인다
2. **계획서의 ✅/⬜ 상태를 직접 읽어서 정확히 파악한다** — 기억에 의존하지 않는다
3. **git 명령어 결과를 직접 확인한다** — 추측하지 않는다
4. **사용자 승인 없이 다음 작업을 바로 시작하지 않는다** — 반드시 '바로 이어서 진행할까요?' 를 먼저 물어본다
5. **실계좌(paper_trading=False) 관련 작업은 별도 확인 없이 절대 진행하지 않는다**

---

## 사용 예시

```
사용자: /ls-catch
```

Claude가 하는 일:
1. `.claude/plans/` 폴더에서 계획서 파일 찾기
2. `git log`, `git status`, `git diff HEAD` 실행
3. 현황(지금까지 한 일 / 수정된 파일 / 최근 커밋 / 다음 할 일)을 쉬운 말로 보고
4. 이어서 진행할지 사용자에게 확인

```
사용자: /ls-catch
Claude: [현황 보고]
사용자: 네
Claude: P4-1 단위 테스트 시작합니다! ...
```
