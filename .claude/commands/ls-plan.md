# ls-plan: 단계별 계획서 작성 커맨드

사용자가 요청한 작업을 체계적으로 분석하고, 단계마다 보고·승인을 받으며 진행한다.
코딩을 모르는 사용자도 완전히 이해할 수 있도록 쉬운 말로 설명한다.

---

## 계획서 파일 생성 규칙

- **파일명 형식**: `YYYYMMDD-계획이름.md` (예: `20260412-골든크로스전략.md`)
- **저장 위치**: 현재 작업 폴더 안 `.claude/plans/` 폴더
- 커맨드 실행 즉시 계획서 파일을 먼저 생성한 뒤 작업을 시작한다

---

## 진행 상태 표시 규칙

모든 단계에 아래 이모지로 현재 상태를 표시한다:

- ⬜ 아직 시작 안 함
- 🔄 지금 진행 중
- ✅ 완료
- ⏸️ 사용자 승인 대기 중

---

## 보고 및 승인 규칙 (매우 중요)

**모든 단계가 끝날 때마다 반드시 멈추고 사용자에게 보고한 뒤 승인을 받는다.**
승인 없이 다음 단계로 절대 넘어가지 않는다.

보고 형식:
```
✅ [단계 이름] 완료 보고

[단계에서 한 일을 쉬운 말로 3~5줄 요약]

👉 다음 단계: [다음 단계 이름]
계속 진행할까요? (네 / 아니오)
```

## PHASE 1 — 자료 수집 및 검증

### P1-1 자료 수집 ⬜

> 목적: 작업에 필요한 파일, 코드, API 정보를 빠짐없이 모은다.

#### 확인 항목
- [ ] 현재 폴더에 있는 파일 목록 파악
- [ ] 관련 코드 파일 내용 읽기
- [ ] 사용 중인 라이브러리(programgarden-finance 등) 기능 확인
- [ ] .env 환경변수 항목 확인
- [ ] 추가로 필요한 외부 정보 파악

#### 수집 결과
(진행 후 채워짐)

---

### P1-2 자료 검증 ⬜

> 목적: 모은 자료가 정확한지 확인한다. 잘못된 정보가 섞여 있으면 나중에 코드가 오작동한다.

#### 확인 항목
- [ ] 수집한 파일 경로와 내용이 실제와 일치하는지 확인
- [ ] API 함수명·파라미터가 최신 라이브러리와 맞는지 확인
- [ ] 코드 간 의존 관계(A가 B를 필요로 하는 관계) 정확히 파악
- [ ] 모의투자/실계좌 관련 설정 확인

#### 검증 결과
(진행 후 채워짐)

#### 확인 항목

### P4-1 단위 테스트 ⬜

> 목적: 각 기능(부품)이 혼자서 제대로 작동하는지 하나씩 확인한다.
> (자동차 부품을 조립 전에 각각 따로 점검하는 것과 같다)
> **반드시 모의투자(paper_trading=True)로만 테스트한다.**

#### 테스트 항목

- [ ] while/for 루프에 종료 조건이 명확히 있는지 확인
- [ ] API 호출 실패 시 무한 재시도하지 않는지 확인
- [ ] 타임아웃(일정 시간 안에 안 되면 포기) 설정 확인

- [ ] 예상치 못한 상황(API 오류, 네트워크 끊김 등)에서 안전하게 종료되는지 확인
- [ ] 로그(기록)가 정상적으로 남는지 확인

#### 테스트 결과
(진행 후 채워짐)

## 절대 규칙

1. **승인 없이 다음 단계로 넘어가지 않는다**
2. **실계좌(paper_trading=False) 관련 코드는 사용자가 명시적으로 요청하기 전까지 절대 추가하지 않는다**
3. **무한루프가 생길 수 있는 코드는 반드시 종료 조건과 타임아웃을 함께 작성한다**
4. **모르거나 불확실한 것은 억측하지 말고 사용자에게 물어본다**
5. **설명은 항상 쉬운 말로 한다 — 전문 용어를 써야 할 때는 바로 옆에 쉬운 설명을 덧붙인다**
6. **문서 업데이트는 반드시 P5-1 최종 승인 이후에만 진행한다**
