## 🎯 목표

`UPBIT_PAPER_TRADING` 플래그에 따라 **모의투자와 실투자의 데이터·기록 영역을 완전히 분리**한다.
모의투자 상태에서 매수·매도를 반복해도 실계좌 데이터(시트·차트·잔고)에는 일체 영향이 없고,
실투자로 전환한 이후에도 **과거 모의 내역이 실제 성과 차트에 섞이지 않도록** 한다.

**기존 문제:**
- 모의 주문이 실행되어도 `mock_balances.json` 이 갱신되지 않아 `get_portfolio_summary()` 가 항상 초기 잔고만 반환.
- `'포트폴리오 추이'` / `'Cumulative PnL'` / 차트가 모드 구분 없이 공용 시트를 사용 → 모의 기간 데이터가 실투자 성과에 섞임.
- `daily_snapshot.json` 이 날짜만 저장해 모드 전환 당일 재기록이 불가능.

**Upbit 환경 특이사항:**
- Upbit 은 paper trading 계좌를 제공하지 않음 → `_paper_trading` 플래그로 주문 실행을 차단하고 `mock_balances.json` 으로 가짜 잔고를 저장하는 내부 구현.
- 시장가 주문이 기본이라 체결가는 `pyupbit.get_current_price()` 로 시뮬레이션.
- 수수료(0.05%)·슬리피지는 모의 모드에서 **반영하지 않음** (전략 단위 검증 목적).

## 배경 분석

### 기존 흐름 (분리 前)

```
[모의투자] place_order(BUY, KRW-BTC, 500,000원)
    ↓
(응답만 흉내)   return {"success": True, ...}
    ↓
mock_balances.json 변경 없음  ← 문제 ①
    ↓
get_portfolio_summary() → 초기 잔고만 반환
    ↓
'포트폴리오 추이' 시트 → 실투자와 공용   ← 문제 ②
'Cumulative PnL' 시트 → 실투자와 공용   ← 문제 ②
```

### 분리 後 목표 흐름

```
[모의투자] place_order(BUY, KRW-BTC, 500,000원)
    ↓
_apply_mock_fill() → mock_balances.json 갱신 (KRW 차감, 코인 추가, 가중평균 업데이트)
    ↓
get_portfolio_summary() → 매수 반영된 실시간 잔고 반환
    ↓
trade_ledger.record_portfolio_snapshot()
    ↓
is_paper_trading()=True → '포트폴리오 추이 (모의)' 시트에 기록
pnl_chart.run_pnl_chart()
    ↓
is_paper_trading()=True → 'Cumulative PnL (모의)' + '실현 손익 누적 차트 (모의)' 생성
```

### 모드 구분 기준

모든 분기는 기존에 존재하던 `upbit_client.is_paper_trading()` 플래그 1개에 의존.
`.env` 의 `UPBIT_PAPER_TRADING=True/False` 로 결정됨.

| 대상 | 실투자 이름 | 모의투자 이름 |
|------|-------------|---------------|
| 포트폴리오 추이 시트 | `포트폴리오 추이` | `포트폴리오 추이 (모의)` |
| PnL 데이터 시트 | `Cumulative PnL` | `Cumulative PnL (모의)` |
| PnL 차트 제목 | `실현 손익 누적 차트` | `실현 손익 누적 차트 (모의)` |
| daily_snapshot.json 의 `mode` 필드 | `"real"` | `"mock"` |

**'시트1' (체결 원장)** 은 `SHEET_HEADERS` 에 이미 `계좌` 열이 존재해 수동으로 구분 가능하므로 분리하지 않음 (대신 향후 `"모의/실전"` 태그를 해당 열에 주입하는 것을 고려할 수 있으나 이번 범위 외).

## PHASE 1 — 자료 수집 및 검증

### P1-1 자료 수집 ✅

#### 수집 결과

- `upbit_client._load_mock_balances()` 이미 존재 → 읽기는 구현돼 있으나 **쓰기 함수 부재**.
- `place_order()` 의 `_paper_trading=True` 분기는 `pyupbit.get_current_price()` 로 모의 체결가만 계산 후 응답을 반환, 파일은 건드리지 않음.
- `trade_ledger.PORTFOLIO_SHEET_NAME = "포트폴리오 추이"` 단일 상수, 422·424·425 행에서 직접 사용.
- `pnl_chart.SOURCE_SHEET_NAME`, `PNL_SHEET_NAME`, `CHART_TITLE` 각 상수 1개씩 → 여러 위치에서 하드코딩.
- `daily_snapshot.json` 구조: `{"last_recorded_date": "YYYY-MM-DD"}` — 모드 구분 없음.
- 하루 1회 가드는 스킵 방식(이미 기록됐으면 `return`) — upsert 가 아니라 완전 스킵.

#### 호출·의존 관계

- `trade_ledger` → `telegram_alert` 만 import (upbit_client ↔ trade_ledger 순환 없음).
- `pnl_chart` → stdlib·gspread 만 (upbit_client 독립적으로 import 가능).
- 따라서 `is_paper_trading()` 을 **지연 import** 로 참조해도 안전.

---

### P1-2 자료 검증 ✅

#### 검증 결과

- 모의 주문 성공 응답의 `executed_volume` / `executed_price` 값을 그대로 `_apply_mock_fill()` 에 넘기면 잔고 갱신 일관성 확보 가능.
- 수량은 `f"{value:.8f}"` 문자열 형식을 유지해야 Upbit 원본 API 응답 포맷과 동일 (pyupbit 내부에서 float 파싱).
- 가중평균 단가 공식: `(old_avg × old_vol + fill_price × fill_vol) / (old_vol + fill_vol)`.
- 전량 매도 시 항목을 **제거** 해야 `holdings_names` 에 "XRP(-100.00%)" 같은 잔재가 남지 않음.

---

### P1-3 부족한 부분 확인 및 보완 ✅

#### 보완 결과

- **마이그레이션 위험**: 기존 `daily_snapshot.json` 에는 `mode` 필드가 없음. 새 로직이 `snapshot_data.get("mode") == current_mode` 로 비교하면 `None != "mock"` 이라 같은 날 재기록이 일어남 — 시트에 "포트폴리오 추이 (모의)" 행 1개가 새로 생김. **의도한 동작** 이므로 문제없음.
- **기존 오염 데이터**: 모의 기간에 누적된 `'포트폴리오 추이'` 시트 데이터는 실투자 전환 후에도 그대로 남아 있음 → 사용자에게 **수동 삭제 권장** 으로 문서화.
- **가드 제거 리스크**: 하루 1회 가드를 유지하되, **같은 날이라도 모드가 다르면 통과** 시켜야 모드 전환 당일에도 올바른 시트에 기록됨.
- **upsert vs 스킵**: `run_daily.py` 가 실패 후 재실행되는 경우 중복 행이 쌓일 수 있음 → upsert 도입이 더 안전(추가 API 호출 1회 `col_values(1)` 만으로 해결 가능).

## PHASE 2 — 설계

### P2-1 설계안 작성 ✅

#### 설계안

**(a) `upbit_client.py` — 모의 잔고 자동 동기화**

| 함수 | 역할 |
|------|------|
| `_save_mock_balances(balances: list) -> bool` | 원자적 쓰기(`.tmp` → `os.replace`) 로 `mock_balances.json` 덮어쓰기 |
| `_apply_mock_fill(side, ticker, exec_vol, exec_price, krw_amount)` | 체결 결과를 잔고에 반영. BUY = KRW 차감·코인 증가·가중평균 갱신. SELL = 코인 차감·KRW 증가·잔량 0 시 항목 제거 |

`place_order()` 의 모의 분기 마지막에 `_apply_mock_fill()` 호출 1줄 추가.
**수수료·슬리피지는 의도적으로 무시** (모의 = 순수 전략 검증).

**(b) `trade_ledger.py` — 모드별 시트 분기**

| 요소 | 변경 |
|------|------|
| `PORTFOLIO_SHEET_NAME` 단일 상수 | `PORTFOLIO_SHEET_NAME_REAL`, `PORTFOLIO_SHEET_NAME_MOCK` 두 상수로 분리 |
| `_portfolio_sheet_name() -> str` 신규 | 런타임에 `upbit_client.is_paper_trading()` 확인해 이름 반환 |
| `_current_mode() -> str` 신규 | `"mock"` 또는 `"real"` 반환 — `daily_snapshot` 저장용 |
| `record_portfolio_snapshot()` | upsert 방식으로 재작성 (오늘 행 존재 시 덮어쓰기) + `mode` 필드 병행 기록 |

**(c) `pnl_chart.py` — 모드별 시트·차트 분기**

| 요소 | 변경 |
|------|------|
| `SOURCE_SHEET_NAME` / `PNL_SHEET_NAME` / `CHART_TITLE` 단일 상수 | 각각 `_REAL` / `_MOCK` 버전으로 분리 |
| `_is_paper()` / `_source_sheet_name()` / `_pnl_sheet_name()` / `_chart_title()` 신규 | 런타임 분기 헬퍼 |
| `update_pnl_worksheet()`, `embed_line_chart()`, `run_pnl_chart()` | 하드코딩된 상수 호출을 헬퍼 함수 호출로 교체. 차트 부제에도 소스 시트명 반영 |

**(d) `daily_snapshot.json` 스키마 확장**

기존: `{"last_recorded_date": "2026-04-21"}`
변경: `{"last_recorded_date": "2026-04-21", "mode": "mock"}`

가드 판정 로직:
```python
if (snapshot["last_recorded_date"] == today
        and snapshot["mode"] == current_mode):
    # 같은 날 & 같은 모드 → upsert 대상
    덮어쓰기
else:
    # 날짜 다름 OR 모드 전환됨 → 신규 append
    새 행 추가
```

### P2-2 설계안 검증 ✅

#### 검증 결과

- **순환 import 회피**: `trade_ledger` 와 `pnl_chart` 의 `_is_paper()` 헬퍼는 함수 내부에서 `import upbit_client` 를 지연 수행. 모듈 최상단 import 하지 않아 circular import 없음.
- **API 호출 비용**: upsert 도입으로 `ws.col_values(1)` 1회 + `ws.update(...)` 1회 = 기존 `ws.append_row()` 1회 대비 +1회. 하루 1회 호출이라 경미.
- **마이그레이션**: 기존 모의 데이터가 `'포트폴리오 추이'` 에 쌓여 있어도 새 시트(`'포트폴리오 추이 (모의)'`) 에 재기록되므로 실투자 시트는 자동으로 깨끗해짐.
- **하위 호환**: 기존 `daily_snapshot.json` 에 `mode` 필드가 없어도 `.get("mode") != current_mode` 로 판정돼 재기록이 일어남(의도된 마이그레이션).
- **테스트 격리**: `mock_balances.json` 을 초기 상태로 되돌리는 스크립트가 필요 → JSON 쓰기로 간단 복원 가능.

### P3-2 구현 ✅

#### 구현 내용

**(a) `upbit_client.py` 모의 잔고 자동 동기화**

| 함수 | 구현 요지 |
|------|-----------|
| `_save_mock_balances(balances)` | `_MOCK_BALANCES_FILE + ".tmp"` 로 쓰고 `os.replace()` 로 원자적 교체. 실패 시 경고 + False. |
| `_apply_mock_fill(side, ticker, exec_vol, exec_price, krw_amount)` | `ticker.split("-", 1)[1]` 로 코인 코드 추출 → `_load_mock_balances()` → KRW/코인 행 업서트 → `_save_mock_balances()`. BUY: KRW ±, 코인 +, 가중평균 단가 재계산. SELL: 코인 −, 1e-12 이하면 항목 제거, KRW +. 알 수 없는 side 는 스킵. 수수료·슬리피지 미반영. |
| `place_order()` 모의 분기 | `_apply_mock_fill()` 호출 1줄 삽입 (try-except 로 감싸 잔고 갱신 실패해도 주문 자체는 성공 응답). |

**(b) `trade_ledger.py` 모드별 시트 분기 + upsert**

| 요소 | 구현 요지 |
|------|-----------|
| `PORTFOLIO_SHEET_NAME_REAL = "포트폴리오 추이"`, `PORTFOLIO_SHEET_NAME_MOCK = "포트폴리오 추이 (모의)"` | 두 개의 상수로 분리. |
| `_portfolio_sheet_name()` | 함수 내부에서 `import upbit_client` 후 `is_paper_trading()` 판정 → 실패 시 실투자 이름(fallback). |
| `_current_mode()` | `"mock" | "real"` 반환. 예외 발생 시 `"real"` fallback. |
| `record_portfolio_snapshot(..., notify_telegram=True)` | (1) mode·sheet_name 계산. (2) A열(`기록시각(KST)`)에서 `today_str` 로 시작하는 행 검색. (3) 있으면 `ws.update(range_name=f"A{r}:J{r}", values=[row])` 로 덮어쓰기 — `op_label="갱신"`. (4) 없으면 `ws.append_row()` — `op_label="신규"`. (5) `_save_daily_snapshot({"last_recorded_date": today_str, "mode": mode})`. (6) 텔레그램은 **신규 등록 시 + notify_telegram=True** 인 경우만 발송 (upsert 스팸 방지). |

**(c) `pnl_chart.py` 모드별 시트·차트 분기**

| 요소 | 구현 요지 |
|------|-----------|
| `SOURCE_SHEET_NAME_REAL/MOCK`, `PNL_SHEET_NAME_REAL/MOCK`, `CHART_TITLE_REAL/MOCK` | 실·모의 이름 상수로 분리. |
| `_is_paper()` | `upbit_client.is_paper_trading()` 지연 import. 예외 시 `False` (실투자 취급). |

## 📝 메모 및 특이사항

- **수수료·슬리피지 미반영** 은 **의도된 설계**. 모의 모드의 목적은 전략 로직 검증이며, 실제 수수료는 Upbit 계좌에서 자동 차감되므로 체결원장(`trade_ledger.json`) 의 `fee` 필드에서 `calc_realized_pnl_total()` 이 이미 반영함. 잔고 계산만 수수료 없이 진행.
- **'시트1'(체결 원장)** 은 분리하지 않음. 모든 매수·매도가 공용 시트에 쌓이지만 `계좌` 열로 수동 필터링 가능. 향후 필요 시 `record_trade()` 에도 mode 주입 가능 (범위 외).
- **`run_all.py` 는 건드리지 않음**. 모의·실투자 분리는 데이터 레이어(`upbit_client`, `trade_ledger`, `pnl_chart`) 에서만 일어나며, 10분 사이클 로직은 동일. 매 사이클에 포트폴리오 추이를 upsert 하려던 초기 설계(STEP 6) 는 **Sheets API 호출 과다** 우려로 철회되고 `run_daily.py` 의 하루 1회 방식이 유지됨.
- **`mock_balances.json` 초기 시드**: 모의투자 시작 시 사용자는 `KRW` 1행만 설정(예: `balance: "10000000"`). 이후 모든 매수·매도는 시스템이 자동 갱신.
- **AWS 배포 시**: 이번 변경으로 `upbit_client.py`, `trade_ledger.py`, `pnl_chart.py` 3개 파일만 업로드하면 됨. `mock_balances.json` 은 초기 1회 세팅 후 건드리지 않음.
- **롤백 절차**: 실투자 모드로 전환 후 다시 모의투자로 돌아가려면 `UPBIT_PAPER_TRADING=True` 로 바꾸고 `rm daily_snapshot.json` 만 하면 자동으로 `(모의)` 시트에 재연결됨.
