## 🎯 목표

현재 진입 조건(동적 목표가 + 30분 가드)에 **오리지널 터틀 트레이딩의 N일 신고가 돌파**를
OR 조건으로 추가한다. 또한 코인 확정 시점에 목표가·기준가를 즉시 저장하고,
보유 코인에도 목표가·기준가를 기록·갱신한다.

**변경 내용 5가지:**
1. **OR 진입 조건 추가** — N일 신고가 돌파 시 30분 가드 없이 즉시 진입 신호 발생
2. **목표가 즉시 저장** — 09:05 코인 확정 시 `pending_target`·`reference_price` 함께 저장
3. **보유 코인 목표가 기록** — 터틀 진입으로 매수된 코인도 목표가·기준가 추적 및 갱신
4. **진입 출처 구분** — `entry_source` 필드로 어떤 조건으로 매수됐는지 기록
5. **보유 코인 감시 우선** — 감시 목록에서 빠진 코인도 `held_coin_record.json` 에 있으면 손절·익절 감시 계속

#### 확인 항목
- [x] N값 결정 — **시스템1: 20일 / 시스템2: 55일 / 둘 다 OR 조건**
- [x] 신고가 기준 — **장중 고가(high) 기준** (일봉 high 필드의 최댓값)
- [x] 진입 신호 반환 형식 — entry_source 필드로 구분 (`"TURTLE_S1"` / `"TURTLE_S2"` / `"TARGET_30MIN"`)

#### 설계 흐름 (초안)

**진입 경로 A — 기존 (동적 목표가 + 30분 가드):**
```
target_manager: pending_target 계산 → 현재가 비교 → above_target_since 기록
timer_agent:    above_target_since 30분 경과 확인 → entry_signals 추가
```

**진입 경로 B — 신규 (오리지널 터틀 N일 신고가 돌파):**
```
target_manager: 현재가 vs 20일 장중고가 비교 → 돌파 시 turtle_s1_signal: true 기록
               현재가 vs 55일 장중고가 비교 → 돌파 시 turtle_s2_signal: true 기록
               (둘 중 하나라도 true 이면 진입 신호)
timer_agent:   turtle_s1_signal 또는 turtle_s2_signal 이 true 인 코인
               → entry_signals 에 합산 (30분 가드 없음)
```

**보유 코인 목표가 기록:**
```
turtle_order_logic: 매수 체결 후 held_coin_record 에 pending_target, reference_price 저장
target_manager:     run_update() 시 held_coin_record 도 함께 갱신
```

### P3-1 indicator_calc.py — N일 신고가 계산 함수 추가 ⬜

추가할 함수:
```python
def calc_n_day_high(ticker: str, n: int) -> float:
    """최근 N일 장중 고가(high) 중 최고값을 반환한다.
    시스템1(n=20), 시스템2(n=55) 모두 이 함수를 재사용한다.
    데이터 부족(n+1개 미만) 시 0.0 반환."""
```

기존 `get_daily_chart(ticker, count=60)` 호출을 재사용하되,
시스템2(55일)는 `count=60` 으로 요청한다 (이미 일봉 60개가 표준).

### P3-4 target_manager.py — 보유 코인 목표가 기록·갱신 로직 추가 ⬜

### P3-5 timer_agent.py — 진입 신호 통합 ⬜

`run_timer_check()` 반환 형식 변경:
- 기존: `["KRW-BTC", "KRW-ETH"]` (티커 리스트)
- 변경: `[{"ticker": "KRW-BTC", "entry_source": "TARGET_30MIN"}, ...]` (딕셔너리 리스트)

수집 순서:
1. `above_target_since` 30분 경과 → `entry_source: "TARGET_30MIN"`
2. `turtle_s1_signal: true` → `entry_source: "TURTLE_S1"` (30분 가드 없음)
3. `turtle_s2_signal: true` → `entry_source: "TURTLE_S2"` (30분 가드 없음)
4. 같은 코인이 중복되면 우선순위: TURTLE_S2 > TURTLE_S1 > TARGET_30MIN 순으로 하나만 포함

`turtle_order_logic.py` 는 `entry_signals` 를 받는 방식만 딕셔너리로 바꾸면
나머지 로직은 그대로 유지된다.

### P3-6 turtle_order_logic.py — entry_source 필드 기록 ⬜

- `signal["ticker"]` 로 티커 접근
- `signal["entry_source"]` 를 `held_coin_record` 에 저장

### P3-7 risk_guardian.py — 보유 코인 우선 감시 ⬜

> 목적: `dynamic_watchlist.json` 에서 빠진 코인이라도 `held_coin_record.json` 에 기록된
> 자동매수 보유 코인은 손절·익절 감시를 계속하도록 수정한다.

#### 현재 동작 (문제)

```python
# 감시 코인이 아니면 건너뜀
if ticker not in watchlist:
    print(f"[risk_guardian] {ticker} 감시 코인 외 → 감시 스킵")
    continue
```

#### 변경 후 동작

```
보유 코인 순회 시:
  1. held_coin_record 에 있으면 → 감시 목록 여부 무관하게 손절·익절 감시 실행
  2. held_coin_record 에 없고 감시 목록에도 없으면 → 수동 보유로 판단, 스킵
  3. held_coin_record 에 없고 감시 목록에 있으면 → 기존과 동일하게 스킵(기준값 없음)
```

**코인명 처리:** `watchlist` 에 없는 경우 티커를 이름 대신 표시한다.
`name = watchlist.get(ticker, {}).get("name", ticker)` 는 이미 이 방식이므로 재사용.
