## 🎯 목표

현재 5~15분마다 `run_all.py`가 실행될 때마다 ATR(변동폭 지표)을 새로 계산하기 위해
Upbit에 차트 데이터를 계속 요청하고 있다.
ATR은 하루 단위 데이터로 계산되므로 하루에 한 번만 계산해도 충분하다.
→ 하루 1회만 계산 결과를 파일에 저장해 두고, 이후 실행에서는 저장된 값을 그대로 쓰는 방식으로 변경한다.
→ Upbit API 호출 횟수를 크게 줄일 수 있다.

**ATR 계산 위치:**
- `indicator_calc.py` → `get_all_indicators(ticker)` 함수
  - 일봉 60개 (`upbit_client.get_daily_chart`) → ATR, 5MA, 20MA, 10일 신저가 계산
  - 240분봉 25개 (`upbit_client.get_minute_chart`) → 240분봉 20MA 계산
  - 즉, 코인 1개당 API 호출 2번 발생

**ATR 사용 위치 (3곳):**
1. `turtle_order_logic.py` line 492 — 신규 진입 시
2. `turtle_order_logic.py` line 548 — 피라미딩 시
3. `risk_guardian.py` line 223 — 손절·익절 감시 시

**현재 문제:**
- `run_all.py`가 5~15분마다 실행될 때마다 ATR을 새로 계산
- ATR은 **일봉(하루 단위) 데이터**로 계산 → 하루 중 수십 번 같은 값을 반복 계산
- 코인 7개 × API 2번 × (24시간/5분) ≈ 하루 4,032번 불필요한 일봉 API 호출

**기존 하루 1회 캐싱 패턴:**
- `daily_snapshot.json` — 날짜 문자열 비교로 하루 1회만 실행하는 패턴 이미 존재

#### 검증 결과

**P1-1 에서 ATR 사용처를 3곳으로 파악했으나 실제로는 4곳이었음:**

| 파일 | 줄 | 호출 상황 |
|------|-----|-----------|
| `target_manager.py` | 182 | 미보유 코인마다 (run 마다 최대 7번) |
| `turtle_order_logic.py` | 492 | 신규 진입 코인마다 |
| `turtle_order_logic.py` | 548 | 피라미딩 후보 코인마다 |
| `risk_guardian.py` | 223 | 보유 코인마다 |

**추가 발견 — `target_manager.py` 에서 일봉 데이터를 두 번 요청하는 중복이 있음:**
- line 182: `get_all_indicators(ticker)` → 내부에서 일봉 60개 요청 (ATR 등 계산)
- line 186: `get_daily_chart(ticker, count=60)` → **같은 일봉 데이터를 한 번 더** 요청 (S1/S2 신고가 계산용)
- → 이 중복은 이번 작업 범위 밖이므로 메모만 해둠

**ATR, 5MA, 20MA, 10일 신저가는 모두 같은 일봉 API 호출 하나에서 나옴:**
- 캐시 파일에 이 4가지를 함께 저장하면, 일봉 API 호출 1번을 아낄 수 있음
- `ma240_20` (240분봉 20MA)은 4시간마다 바뀌는 값이므로 매번 새로 가져와야 함

**`daily_snapshot.json` 패턴 확인:**
```json
{"last_recorded_date": "2026-04-22", "mode": "mock"}
```
- KST 날짜 문자열 비교로 하루 1회 제한 → 동일 패턴 적용 가능

## PHASE 2 — 설계

### P2-1 설계안 작성 ⬜

> 목적: 어떤 파일에 어떤 기능을 넣을지 설계도를 그린다.

#### 설계안

---

**1. `atr_cache.json` 파일 구조 (스키마)**

코인별로 날짜와 지표값을 저장한다.
날짜가 오늘과 다르면 "오래된 캐시"로 판단해 새로 계산한다.

```json
{
  "KRW-BTC": {
    "date":      "2026-04-22",
    "atr":       1234567.0,
    "ma5":       93000000.0,
    "ma20":      91000000.0,
    "day10_low": 88000000.0
  },
  "KRW-ETH": {
    "date":      "2026-04-22",
    "atr":       45000.0,
    "ma5":       1800000.0,
    "ma20":      1750000.0,
    "day10_low": 1700000.0
  }
}
```

---

**2. `indicator_calc.py` 수정 내용 (이 파일 하나만 바꿈)**

현재 `get_all_indicators()` 함수 흐름:
```
① 일봉 60개 API 호출
② 240분봉 25개 API 호출
③ ATR, 5MA, 20MA, 10일 신저가, 240분봉 20MA 계산
④ 결과 반환
```

수정 후 흐름:
```
① atr_cache.json 읽기
② 오늘 날짜 캐시가 있으면?
   → ATR, 5MA, 20MA, 10일 신저가 = 캐시에서 가져옴
   → 240분봉 25개 API 호출 (ma240_20만 새로 계산)
③ 캐시가 없거나 날짜가 다르면?
   → 일봉 60개 API 호출
   → 240분봉 25개 API 호출
   → 모든 지표 계산
   → atr_cache.json에 오늘 날짜로 저장
④ 결과 반환
```

---

**3. 새로 추가할 내부 함수 2개 (`indicator_calc.py` 내부에만 존재)**

```
_load_atr_cache()  → atr_cache.json 읽기
_save_atr_cache()  → atr_cache.json 쓰기
```

함수 이름 앞 `_`는 "이 파일 안에서만 쓰는 함수"라는 약속이다.

---

**4. 데이터 흐름 요약**

```
run_all.py (5분마다 실행)
  └─ risk_guardian / turtle_order_logic / target_manager
       └─ indicator_calc.get_all_indicators("KRW-BTC")
            ├─ [캐시 있음] atr_cache.json 읽기 → ATR 등 즉시 반환
            │             + 240분봉 API 1회만 호출
            └─ [캐시 없음] 일봉 API + 240분봉 API 호출
                          계산 후 atr_cache.json 저장 → 반환
```

---

**5. API 호출 절감 효과**

| 구분 | 코인 1개당 하루 호출 수 |
|------|----------------------|
| 변경 전 | 일봉 1회 × 288회 실행(5분 간격) = **288번** |
| 변경 후 | 일봉 **1번** (첫 실행에만) |
| 절감 | 일봉 API **287번 감소** (코인당) |
| 7개 코인 합산 | 하루 약 **2,009번** 감소 |

### P2-2 설계안 검증 ⬜

#### 확인 항목
- [ ] 설계가 목표를 완전히 달성하는지 확인
- [ ] 기존 코드 흐름을 최소한으로 바꾸는지 확인
- [ ] 실계좌 관련 위험 요소 없는지 확인

#### 검증 결과

**1. 설계 목표 달성 여부 ✅**
- 일봉 API 호출을 하루 1회로 줄이는 목표 달성 가능

**2. 추가 발견 — `get_n_day_high_signals()` 함수는 사용되지 않음**
- `indicator_calc.py`에 정의되어 있으나 어느 파일에서도 호출하지 않음
- `target_manager.py`가 자체적으로 `get_daily_chart()`를 호출해 신고가를 계산 중
- 이번 작업과 무관하므로 건드리지 않음

**3. `.gitignore` 누락 — 반드시 추가 필요**
- `atr_cache.json`이 `.gitignore`에 없음
- 런타임 생성 파일이므로 커밋 방지를 위해 추가해야 함

**4. 예외 상황별 동작 검증**

| 상황 | 동작 | 안전성 |
|------|------|--------|
| 캐시 파일 없음 (처음 실행) | 캐시 미스 → API 호출 → 저장 | ✅ |
| 같은 날 두 번째 실행 | 날짜 일치 → 캐시 사용 | ✅ |
| 날짜가 바뀐 후 첫 실행 | 날짜 불일치 → 재계산 → 저장 | ✅ |
| 캐시 파일 JSON 손상 | 읽기 오류 → 빈 딕셔너리 → 재계산 | ✅ |
| API 실패 (일봉 못 가져옴) | 0.0 반환, 캐시 저장 안 함 → 다음 실행에 재시도 | ✅ |
| 특정 코인만 API 실패 | 해당 코인만 영향, 다른 코인 무관 | ✅ |

**5. 범위 밖 추가 최적화 기회 (이번에 안 함)**
- `target_manager.py` line 186의 일봉 중복 호출도 줄이면 추가 절감 가능
- 다음 작업 기회에 검토 권장

### P2-3 프로젝트(코드) 구조 작성 ✅

#### 최종 파일 구조
```
indicator_calc.py   ← 수정 (캐시 로직 추가)
.gitignore          ← 수정 (atr_cache.json 한 줄 추가)
atr_cache.json      ← 런타임 자동 생성 (커밋 금지)
```

#### 각 파일 역할 설명

**`indicator_calc.py` 변경 내용:**
- ATR_CACHE_FILE 경로 상수 추가
- `_load_atr_cache()` 내부 함수 추가 — 캐시 파일 읽기

- `get_all_indicators()` 수정 — 캐시 확인 → 유효하면 일봉 API 생략

**수정 전/후 핵심 차이:**
```
[수정 전] 매 실행마다: 일봉 API + 240분봉 API = 2번
[수정 후] 첫 실행:     일봉 API + 240분봉 API = 2번 → 캐시 저장
          이후 실행:   240분봉 API = 1번 (일봉은 캐시에서)
```
