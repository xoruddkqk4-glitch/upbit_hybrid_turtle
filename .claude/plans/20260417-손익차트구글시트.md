## 🎯 목표

구글 스프레드시트에 기록된 하루 단위 실현 손익 데이터를 읽어서,
새로운 시트에 날짜별 누적 손익 차트를 자동으로 그린다.
자본금 추가·인출(KRW 입출금)은 반영하지 않고, 실현 손익만 누적해서 시각화한다.

**Upbit 환경 특이사항:**
- 체결 원장(trade_ledger) 의 SELL 기록에는 `profit_rate` 필드가 이미 포함되어 있음
- 실현 손익 금액 = `(매도 단가 - 평균 매입단가) × volume - fee`
- 암호화폐는 소수점 수량이라 `volume × price` 계산 시 반올림 처리 필요

## PHASE 1 — 자료 수집 및 검증

### P1-1 자료 수집 ⬜

#### 수집 결과
- `trade_ledger.py` 의 `SHEET_HEADERS` 에 `profit_rate`, `net_amount` 필드 이미 존재
- Upbit 체결 원장 SELL 행의 `net_amount` = 매도대금 - 수수료
- 실현손익 = SELL 행의 `net_amount` - (`volume` × 당시 `avg_buy_price`)
- gspread 는 `worksheet.insert_chart()` 를 지원하지 않으므로 구글 시트 자체 차트 생성 API (`Sheets API v4 - batchUpdate`) 활용

---

### P1-2 자료 검증 ⬜

#### 검증 결과
- gspread 의 `spreadsheets.batchUpdate` 를 통해 `EmbeddedChart` 추가 가능
- Upbit 실현 손익 계산 시 `profit_rate` 가 음수일 수 있으므로 손실·수익 모두 정확히 반영되도록 구현
- KRW 입출금 이벤트는 체결 원장에 기록되지 않으므로 자동 제외됨

### P1-3 부족한 부분 확인 및 보완 ⬜

#### 보완 결과
- 매매 수수료(`fee`) 필드를 반드시 차감해야 정확한 실현 손익이 됨
- 같은 날 여러 번 매도된 경우 → 하루 단위로 합산

## PHASE 2 — 설계

### P2-1 설계안 작성 ⬜

#### 설계안

**새로 만들 파일: `pnl_chart.py`**

| 함수 | 역할 |
|------|------|
| `build_pnl_series()` | 체결 원장 SELL 행을 읽어 날짜별 실현 손익 시계열 생성 |
| `update_pnl_worksheet(series)` | `Cumulative PnL` 워크시트에 날짜·일일손익·누적손익 3열 갱신 |
| `embed_line_chart()` | 해당 워크시트에 선 그래프 차트를 삽입 (Sheets API batchUpdate) |
| `run_pnl_chart()` | 위 3단계를 순서대로 실행하는 진입점 |

**워크시트 스키마 (`Cumulative PnL`):**

| 열 | 내용 |
|----|------|
| A | 날짜 (YYYY-MM-DD) |
| B | 당일 실현 손익 (KRW) |
| C | 누적 실현 손익 (KRW) |
| D | 매매 코인 수 |

### P2-2 설계안 검증 ⬜

#### 검증 결과
- 체결 원장 SELL 행을 필터해 날짜별로 합산하는 방식은 O(n) 이라 원장 수천 행도 무리 없음
- 차트 삽입은 최초 1회만 생성, 이후 업데이트는 데이터만 갱신

### P3-2 구현 ✅

#### 구현 내용

**신규 파일**: `upbit_hybrid_turtle/pnl_chart.py`

| 함수 | 구현 요지 |
|------|-----------|
| `_load_ledger_rows()` | `trade_ledger.json` 을 읽어 `ts_kst` 오름차순 정렬. 파싱 실패 행은 뒤로 밀리고 list/dict 가드 포함. |
| `build_pnl_series(rows)` | 코인별 러닝 포지션(`qty`, `avg_cost`) 을 시간 순서대로 추적. BUY 는 가중평균 매입단가 재계산, SELL 은 `(unit_price − avg_cost) × sell_qty − fee` 를 당일 손익에 가산. KRW 입출금은 원장에 없으므로 자동 제외. |
| `update_pnl_worksheet(series)` | `Cumulative PnL` 워크시트 `clear()` 후 헤더 + `날짜 / 당일 실현손익 / 누적 실현손익 / 매매 코인 수` 4열을 `A1` 부터 일괄 갱신(`value_input_option="RAW"`). 워크시트 없으면 1000×10 크기로 신규 생성. |
| `_chart_exists()` | `spreadsheet.fetch_sheet_metadata()` 로 같은 제목의 `EmbeddedChart` 존재 여부 확인 → 재생성 스킵 기준. |
| `embed_line_chart()` | Sheets API `batch_update` 로 `addChart` 요청. domain=A열(날짜), series=B/C열(당일·누적 실현손익), 범위는 `startRowIndex=0` 만 지정(종료 미지정 → 행 추가 시 자동 반영). F열 1행을 앵커로 640×380 px 고정 위치. **최초 1회만 삽입**. |
| `run_pnl_chart()` | 원장 로드 → 시리즈 생성 → 시트 갱신 → 차트 임베드 순. 원장이 비었거나 SELL 이 없으면 조기 종료. |

| 날짜 | 거래 | 기대 손익 | 비고 |
|------|------|-----------|------|
| 2026-04-01 | BUY 0.01 BTC @100M | — | avg_cost=100M, qty=0.01 |
| 2026-04-01 | BUY 0.01 BTC @110M (피라미딩) | — | avg_cost=105M(가중평균), qty=0.02 |
| 2026-04-02 | SELL 0.005 BTC @120M (부분 매도) | **+75,000원** | (120M−105M)×0.005 |
| 2026-04-02 | BUY 1,000 XLM @200, fee 100 | **−100원** | 수수료만 손익 반영 |
| 2026-04-02 | SELL 1,000 XLM @220, fee 100 | **+19,900원** | (220−200)×1000 − 100 |
| 2026-04-03 | SELL 0.015 BTC @90M (손절) | **−225,000원** | (90M−105M)×0.015 |

**실행 결과**:
```
2026-04-02  daily=+94,800원  cumulative=+94,800원  trade_count=2
2026-04-03  daily=-225,000원 cumulative=-130,200원 trade_count=1
```

수기 계산(94,800 = 75,000 − 100 + 19,900) 및 누적(−130,200) 과 **완전 일치**.

```
[pnl_chart] 'Cumulative PnL' 워크시트 생성
[pnl_chart] 'Cumulative PnL' 데이터 갱신 완료 (3일)
[pnl_chart] '실현 손익 누적 차트' 라인 차트 임베드 완료

```

누적 +7,500원 = (+10,000 XLM 부분익절) + (−20,000 BTC 손절) + (+17,500 XLM 잔량 익절). 수기 검산과 일치.

**Idempotent 검증** (동일 입력 2회 호출):
```
--- 1회차 ---  [pnl_chart] '실현 손익 누적 차트' 이미 존재 → 차트 재생성 스킵
--- 2회차 ---  [pnl_chart] '실현 손익 누적 차트' 이미 존재 → 차트 재생성 스킵
```

**테스트 후 정리**: 데모 SELL 5건은 `trade_ledger.json.bak` 백업→복구로 원장 오염 없음. `Cumulative PnL` 시트도 헤더만 남기고 정리 완료.

## 📝 메모 및 특이사항

- 차트 형식: 날짜별 누적 실현 손익 선 그래프 (x축 = 날짜, y축 = 누적 KRW 손익)
- KRW 입출금(자본금 추가·인출) 은 체결 원장에 기록되지 않으므로 자동 제외
- 수수료(`fee`) 는 반드시 차감해서 순손익을 계산
- Upbit 는 암호화폐 소수점 수량이라 volume × price 계산 시 반올림에 유의
- gspread 는 차트 삽입을 직접 지원하지 않으므로 Sheets API batchUpdate 로 `EmbeddedChart` 추가
