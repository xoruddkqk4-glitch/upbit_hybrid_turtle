## 배경 분석

### 문제 상황

자동매매 도중 수동으로 일부 코인을 팔거나 사면
(Upbit 앱에서 손절해버리거나 추가 매수),
`held_coin_record.json`(시스템이 기억하는 보유 현황) 과
실제 Upbit 계좌 잔고가 어긋나게 된다.

이 상태로 자동매매가 계속 돌면:
- 이미 판 코인에 손절 주문을 또 넣으려 할 수 있음
- 피라미딩 수량 계산이 틀어짐
- 감시해야 할 코인을 빠뜨림

### 동기화 기준

**실제 잔고를 기준(진실)으로** `held_coin_record.json` 을 맞춘다.

| 상황 | 처리 방법 |
|------|-----------|
| 기록엔 있는데 실제로 없음 | 기록에서 삭제 + 텔레그램 알림 |
| 실제로 있는데 기록에 없음 | 알림만 발송 (터틀 파라미터 불명 — 자동 추가 안 함) |
| 둘 다 있는데 수량이 다름 | 실제 수량으로 기록 업데이트 + 텔레그램 알림 |
| 일치 | 아무것도 안 함 |

### "실제로 있는데 기록에 없음" — 자동 추가를 안 하는 이유

터틀 전략은 `stop_loss_price`(손절가), `next_pyramid_price`(피라미딩 기준가),
`current_unit` 등의 파라미터가 있어야 정상 작동한다.
수동 매수 코인은 이 값들을 알 수 없으므로 자동으로 집어넣으면 오히려 더 위험하다.
**알림만 보내고 사용자가 직접 판단하게 한다.**

### 암호화폐 특유 주의 사항

- Upbit 의 `get_balances()` 응답은 `{"currency": "BTC", ...}` 처럼 **통화 코드(BTC) 만** 반환한다.
  → `KRW-BTC` 티커 형태로 변환해서 비교해야 한다.
- Upbit 는 **소수점 수량** 을 허용하므로 수량 비교 시 `abs(실제 - 기록) < 1e-8` 정도의 허용오차를 둔다.
- 수수료·에어드랍으로 인한 미세 잔량(0.00000001 단위)은 `volume < dust_threshold` 기준으로 기록 대상에서 제외한다.

## PHASE 3 — 구현

### P3-1 `balance_sync.py` 신규 파일 작성

**역할:** 실제 잔고 ↔ `held_coin_record.json` 비교·동기화

**함수 1개:**

```python
def run_balance_sync() -> bool:
    """실제 잔고와 held_coin_record.json 을 비교해서 불일치를 수정한다.

    Returns:
        True  — 동기화 성공 (불일치가 없거나 수정 완료)
        False — 잔고 조회 실패 (upbit_client.get_balance() 가 빈 리스트 반환)
    """
```

**처리 흐름:**

```
1. upbit_client.get_balance() 호출
   → 빈 리스트면 False 반환 (API 오류 상황 — 실행 중단 신호)
   → 응답 통화 코드를 KRW-XXX 티커 형태로 변환

2. held_coin_record.json 읽기
   → 파일 없으면 빈 dict 로 처리

3. 비교 및 수정
   ① 기록에 있는데 실제로 없는 코인
      → held_coin_record 에서 해당 티커 삭제
      → 텔레그램: "⚠️ [잔고동기화] {코인명}({티커}) — 실제 잔고 없음. 기록 삭제함."

   ② 실제로 있는데 기록에 없는 코인 (감시 목록 여부 무관)
      → held_coin_record 변경 없음
      → 텔레그램: "⚠️ [잔고동기화] {코인명}({티커}) {volume} — 기록 없는 보유 코인 발견. 수동 확인 필요."

   ③ 둘 다 있는데 수량이 다른 코인
      → held_coin_record 의 total_volume 을 실제 수량으로 교체
      → 텔레그램: "⚠️ [잔고동기화] {코인명}({티커}) 수량 불일치: 기록 {기록수량} → 실제 {실제수량} 으로 수정."

   ④ 수수료·에어드랍 등 먼지 잔량(5000원 미만 평가액) 은 ②·③ 비교 대상에서 제외

4. 변경사항이 1건 이상이면 held_coin_record.json 파일에 저장

5. True 반환
```
