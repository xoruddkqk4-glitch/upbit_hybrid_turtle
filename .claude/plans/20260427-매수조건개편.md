### P2-2 설계안 검증 ✅

#### 검증 결과

**✅ 정상 동작 확인 항목**
- `run_all.py` 실행 순서 변경 없음 — target_manager → timer_agent → turtle_order_logic 순서 유지
- `run_update()` 함수 시그니처 동일 — `run_all.py` 호출부 수정 불필요
- `run_timer_check()` 반환값 형식 동일 — `turtle_order_logic.run_orders()` 수정 불필요
- S1/S2 우선순위(S2 > S1) 로직 그대로 유지
- EXCEPTION 진입 시 `max_unit=2` 는 변경된 `MAX_UNIT_PER_COIN=3` 보다 여전히 작으므로 문제 없음

**⚠️ 발견된 이슈: 기존 파일 호환성**
- 현재 서버에 이미 `unheld_coin_record.json` 가 있을 수 있음
- 기존 파일에는 `turtle_s1_since` / `turtle_s2_since` 필드가 없음
- 만약 기존 파일에서 `turtle_s1_signal: true` 이고 `_since` 가 없으면
  → 새 timer_agent 는 30분 미경과로 판단해 영원히 신호를 내지 않는 문제 발생

**해결책: `_since` 초기화 규칙 추가**
- target_manager 에서 신호 업데이트 시:
  - `True → True` 이고 `_since` 가 None → 현재 시각으로 새로 세팅 (30분 타이머 재시작)
  - `True → True` 이고 `_since` 가 이미 있음 → 유지 (기존 타이머 보존)
- 이렇게 하면 코드 업데이트 직후 첫 실행에서 기존 True 신호도 타이머가 올바르게 시작됨
- 보수적으로 30분을 다시 기다려야 하지만 안전함

**기타 확인 사항**
- 기존 파일의 `pending_target`, `reference_price`, `above_target_since` 필드는
  새 코드가 읽지 않으므로 파일에 남아 있어도 무해 — 다음 실행에서 덮어써짐
- `indicator_calc.py` 의 `ma240_20` 계산은 그대로 남지만 새 코드에서는 사용 안 함
  → 불필요한 API 호출이 약간 남지만 매매 동작에 영향 없음 (이번 범위 외)
