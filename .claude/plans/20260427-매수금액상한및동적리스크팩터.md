## 🎯 목표

현재의 NORMAL/CAPPED/EXCEPTION/SKIP 4단계 분류 방식을 아래와 같이 단순화·개편한다.

1. **1 Unit당 최대 매수 금액 = 총 자본 × 0.1** (10%)
2. 계산된 1 Unit 매수금이 최대 금액 이하 → 그대로 매수
3. 계산된 1 Unit 매수금이 최대 금액 초과 → 해당 종목의 risk factor를 낮춰서
   1 Unit 매수금 = 최대 금액이 되도록 조정
4. **조정된 risk factor는 종목별로 다르며**, 2N 손절가와 함께 `held_coin_record.json`에 저장

#### 수집 결과

**[turtle_order_logic.py] 현재 calc_unit_size() 구조**
- 전역 상수 `RISK_PER_TRADE = 0.01` 사용
- 수식: `risk_volume = (capital × 0.01) / ATR`, `risk_krw = risk_volume × price`
- `tier_ratio = risk_krw / capital` 기준으로 4단계 분류:
  - ≤ 10% → NORMAL: 원본 수량 사용, max_unit=3
  - 10~25% → CAPPED: 자본 10%로 축소, max_unit=3
  - 25~50% → EXCEPTION: 자본 10%로 축소, max_unit=2
  - > 50% → SKIP: 진입 불가
- 반환값: `(volume, krw_amount, max_unit, entry_tier)` 튜플

**[tier 관련 상수] 현재 turtle_order_logic.py 내 정의**
- `RISK_PER_TRADE = 0.01` (53번 줄)
- `NORMAL_TIER_PCT = 0.10` (71번 줄)
- `CAPPED_TIER_PCT = 0.25` (72번 줄)
- `EXCEPTION_TIER_PCT = 0.50` (73번 줄)
- `EXCEPTION_MAX_UNIT = 2` (75번 줄)

**[held_coin_record.json] 현재 저장 필드**
- current_unit, last_buy_price, avg_buy_price, stop_loss_price
- next_pyramid_price, max_unit, total_volume, entry_source, entry_tier
- (balance_sync 수동편입 시) manual=True, max_unit=1 추가

**[다른 모듈의 tier/risk 참조]**
- `risk_guardian.py`: stop_loss_price, avg_buy_price, last_buy_price 만 읽음 → 영향 없음
- `balance_sync.py`: max_unit 필드만 씀 (manual=True 종목에 max_unit=1 설정) → 영향 없음
- `entry_tier` 필드: turtle_order_logic.py 에서만 씀/읽음 → 다른 파일 영향 없음
